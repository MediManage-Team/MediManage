# Subscription AI Fallback Rules Engine Baseline (v1)

Status date: 2026-02-23  
Scope: Deterministic fallback path for checkout when AI care-protocol generation is unavailable.

## 1) Problem Statement

Checkout previously depended on AI care-protocol generation. If AI failed, checkout could be blocked in the asynchronous flow.

## 2) Fallback Strategy

1. `BillingService.generateCheckoutCareProtocol(...)` now attempts AI first.
2. If AI is unavailable, throws, or returns non-actionable output, the flow falls back to deterministic rules output.
3. Fallback output is generated by `BillingCareProtocolFallbackRulesEngine`.

## 3) Continuity Contract

1. Checkout must continue even during AI outage.
2. Fallback protocol includes:
   - generic dose/safety reminders,
   - medicine list snapshot from current bill,
   - explicit AI-unavailable reason.
3. Fallback does not alter billing totals or discount logic.

## 4) Safety Boundaries

1. Fallback text is advisory and generic.
2. It avoids model inference and external API dependency.
3. Final dispense responsibility remains with pharmacist/doctor guidance.

## 5) Validation Coverage

1. `BillingServiceSubscriptionCheckoutTest.generateCheckoutCareProtocolFallsBackToRulesEngineWhenAiUnavailable` validates fallback behavior.
2. Existing checkout tests remain green, confirming no regression in billing path.
